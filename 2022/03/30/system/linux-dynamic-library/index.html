Linux 动态链接库相关整理[WIP]

Posted on 2022-03-30 Edited on 2022-03-31 In System

PingCAP（aka 贵司）自研了套部署工具 TiUP 来取代早前使用的重型部署工具 Ansible。实际使用过程中，有多次出现在打 patch 滚动重启过程中，TiFlash 节点行为不符合预期。后来发现，TiUP 打 patch 过程中，没有停进程且是直接 tar 命令解压覆盖部署目录，最基本的原子性保障更是无从谈起。

v6.0 版本前 TiFlash 的部分模块包含动态加载共享库（aka 动态库）的行为，这类行为会被搜索路径下的文件影响。如果存在某个时刻前后引入的二进制库不兼容，则直接影响程序的行为。

因为是用 tar 命令解压覆盖，反而是没有出现老生常谈的 cp 覆盖动态库引起 Segmentation fault 的问题。

正好在此整理下动态库相关的问题。

Linux 文件系统基础

相关基础参考 Modern Operating Systems，深度细节建议看操作系统代码实现。

Inode

Linux 下 Inode 是一个重要概念，是理解 Unix/Linux 文件系统和硬盘储存的基础。Inode、软链接、硬链接 等相关概念基础参考 阮一峰. 理解inode，解释得比较易懂。以下部分做点补充。

Inode 本身是一种抽象设计，不同操作系统有各自实现，此处针对传统的类 Unix 文件系统。

通过命令获取某块 3.6T 硬盘的 Inode size 值为 256

该硬盘的 Inode 存储上限是 233M。

可以推算出，该硬盘最多存储 954368 个 Inode。如果每个 Inode 对应一个 4K 大小的独立目录，总容量也不超过 3.65G。因此，确实会出现因 Inode 耗尽而无法创建文件的极端情况。

通过 Inode 号码直接 直接删除 Inode 节点，等同于删除文件

Unix/Linux 系统中，目录（directory）也是一种文件：

  • 创建目录时，默认会生成两个目录项：”.” 和 “..”。
  • 前者的 Inode 号码就是当前目录的 Inode 号码，等同于当前目录的 “硬链接”，也就使得新建空目录的 “链接数” 加 1 等于 2。
  • 后者的 Inode 号码就是当前目录的父目录的 Inode 号码，等同于父目录的 “硬链接”，令父目录的 “链接数” 加 1。
  • 对于根目录，则两者都指向自己。

“硬链接” 目标和源因为共用同个 Inode，因此不能跨盘建立，否则报错 Invalid cross-device link。通过 ln ${src} ${tar} 建立 “硬链接” 无法作用于目录，否则报错 hard link not allowed for directory。

与 “硬链接” 不同，”软链接” 自身就是一个独立的二进制文件，每次使用时再由文件系统解析到对应的目标，如果解析失败则会报错。也就意味着 “软链接” 不会对文件系统本身产生侵入性影响。

“硬链接” 自身就是文件系统中的一环，链接数归 0 后系统才会尝试回收对应的文件，与引用计数的管理策略类似。如果目录可以建立 “硬链接”，则容易造成循环引用，导致文件无法回收。

cp/rm/mv/tar 命令

用 strace 可以很清楚地看到命令执行时的系统调用。通常来说 rm 主要用到 unlink*，mv 主要用到 rename*，cp 主要用到 open、read、write。

根据 linux/man-pages/unlink，unlink 用于按照文件名删除文件

  • 如果文件无其他链接，且没有进程打开该文件，则删除文件并释放空间。
  • 如果文件仅此一份链接，但存在进程仍然打开该文件，则删除后文件将一直存在，直到引用它的最后一个文件描述符关闭，系统才会回收。

因此通过 unlink 删除正在被使用的文件是安全的。

rename 面对的场景更加复杂。根据 linux/man-pages/rename

  • 目标文件已经存在时，如果是非跨盘操作，会保证替换的原子性
  • 如果是跨盘操作，则会报错 Invalid cross-device link；mv 命令会忽略报错，通过 unlink 删除目标文件，新建文件后复制内容，最后 unlink 删除源文件。

当目标文件已存在，cp 命令调用 open 的参数会包含 O_TRUNC，即 复用目标文件 Inode 并清空内容，再把源文件内容写入目标文件。

追踪 tar 命令的执行过程，可以看到当目标文件已存在时，是 unlink 删除后再新建并复制写入。所以通过 tar 命令覆盖正在被使用的文件也是安全的。

当程序正在运行时，调用 cp 尝试覆盖其二进制文件，会报错 Text file busy。如果使用 cp -f 则会忽略报错，并执行 unlink 后再复制，此时 Inode 已经发生改变。

Linux 动态库

部分基本概念可参考 linux下静态库、动态库总结。强烈推荐 《程序员的自我修养——链接装载与库》 这本书，细致全面地囊括了各项相关知识。

Linux 下的库有两种：静态库 和 共享库

  • 静态用 .a 为后缀。静态库的代码在编译过程中已经被载入可执行程序，因此生成的可执行程序体积较大。
  • 动态库的代码是在可执行程序运行时才载入内存的，在编译过程中仅简单的引用，因此生成的可执行程序代码体积较小。

Linux 动态库通常用 .so 为后缀，MacOS 为 .dylib，Windows 为 .ddl。

动态库的好处有：

  • 不同的应用程序如果调用相同的库，那么在内存里只需要有一份该共享库的实例
  • 具有部分相对独立的构建环境，减少与宿主进程产生链接符号污染
  • 编译链接代价小，对于长期稳定的模块库，无需反复拷贝数据
  • 库动态替换，可支持程序热更新

使用 ldd 工具，查看可执行程序依赖那些动态库或着动态库依赖于那些动态库。

使用 nm 命令，查看静态库和动态库中有那些函数名：

  • T类：是在库中定义的函数
  • U类：是在库中被调用，但并没有在库中定义（表明需要其他库支持）
  • W类：是所谓的 “弱态” 符号，它们虽然在库中被定义，但是可能被其他库中的同名符号覆盖

替换动态库

如果动态库正在被使用时，执行 cp ${newlib}.so ${oldlib}.so，则容易引起程序 core 掉。原因在于：

  • 应用程序通过打开动态库时，内核通过 mmap 把 so 加载到进程地址空间，对应于 Virtual memory area(VMA) 里的几个 page，可参考 Linux小记. 动态库加载方式
  • 在这个过程中 dynamic linker/loader 会把 so 里面引用的外部符号例如 malloc、printf 等解析成真正的虚存地址
  • 当 so 被 cp 覆盖时，确切地说是被 truncate 时，内核会把 so 文件在虚拟内存的页清除掉
  • 当运行到 so 里面的代码时，因为虚拟内存的页已经清除掉了，这时会产生一次缺页中断
  • 缺页中断会导致内核从 so 文件中拷贝对应的页到内存中去：
    • 如果 so 里面依赖了外部符号，但是这时的外部符号并没有经过重新解析，当调用到时就产生 segment fault
    • 如果需要的文件偏移大于新的 so 的地址范围，就会产生 bus error

如果 so 没有依赖外部符号，则不会 core 掉

为什么系统会阻止 cp 覆盖可执行程序，而不阻止覆盖 so 文件？

  • 操作系统的 Demand Paging 机制下，程序运行时并不会将所有 page 都加载到内存中，而只有在系统有访问需求时才将其加载
  • 要求正在运行中的程序镜像（并非文件本身）不被意外修改，因此内核在启动程序后会锁定这个程序镜像的 Inode。
  • so 文件是靠 ld.so 加载的，属于用户态程序，没有权利去锁定 Inode。

结合上述内容，替换动态库时，必须先执行系统调用 unlink* 删除目标文件。因此禁止直接使用 cp，可选 install、rm + cp 等

动态库链接

动态链接｜执行时搜索路径顺序：

  1. 编译目标代码时指定的动态库搜索路径
  2. 环境变量 LD_LIBRARY_PATH 指定动态库搜索路径
  3. 配置文件 /etc/ld.so.conf 中指定的动态库搜索路径
  4. 默认的动态库搜索路径：/lib、/usr/lib、/usr/local/lib

问题分析

静态链接的过程主要包括：地址和空间分配（Address and Storage Allocation）、符号决议（Symbol Resolution，aka 符号绑定）、重定位（Relocation）等。

上述例子使用动态库的方式为隐式加载（载入时加载）

  • 类似静态库链接的过程发生在程序加载时，动态链接器将所有相关动态库装载到进程地址空间，将程序中未定义的符号绑定到相应的动态链接库，进行重定位工作。
    • 链接器按照深度｜广度优先顺序加载把程序和相关共享库的符号表都合并到全局符号表（Global Symbol Table）中
    • 链接器基本规则：全局符号表中已存在同名符号，则后加入的忽略
  • 首先载入主程序，main 文件中默认暴露了 foo 函数且对应 v2 库的定义，foo 函数符号先注册并被定位到该入口。
  • 由于 libtest.so 对外暴露了 foo 函数，所以 test 函数调用 foo 无法按照模块内函数调用的方，编译器会当作模块外函数处理。
  • 载入 libtest.so 符号表时，foo 函数符号注册失败，最终程序执行 v2 库的逻辑。

交换库链接顺序或者只链接 test 库时，也是同样原理，程序中 foo 函数符号被决议成为未定义（U类），加载 libtest.so 时注册。

解决符号冲突

为避免符号冲突，可通过以下几种方式

显式加载（运行时加载）

程序运行时，在逻辑中加载 libtest.so 获取 test 函数入口。编译时则不需要链接项 -ltest。

当进程中的模块是通过 dlopen() 载入的共享对象，dlsym() 查找符号的优先级分为 2 种：

  • dlopen() 参数 filename 为 nullptr， 则是从全局符号表查找，即 装载序列（Load Ordering）。
  • dlopen() 指定共享对象时，则采用 依赖序列（Dependency Ordering） 的优先级：以被 dlopen() 打开的共享对象为节点，对其依赖对象进行广度优先遍历，直到找到符号。

符号隐藏

隐藏 main 入口 foo 函数

隐藏 test 库 foo 函数（推荐这种方式，尽可能避免对外暴露无关符号）
